## ADR-001: Выбор технологического стека

### Контекст

Необходимо разработать A/B‑платформу, которая должна соответствовать следующим требованиям:

- **Высокая производительность**: обработка тысяч запросов `/decide` с минимальной задержкой
- **Строгая типизация**: критичность ошибок бизнес-логики
- **Ограниченные сроки**: меньше 2-х недель на разработку MVP с полным покрытием функционала
- **Воспроизводимость**: легкость развёртывания и отсутствие скрытых зависимостей

Требуется выбрать язык программирования, веб-фреймворк и инструменты работы с базой данных (ORM/Driver)

### Решение

Выбран следующий стек:

- **Язык**: `Go 1.24+` — обеспечивает высокую производительность (компилируемый язык, горутины) при простоте разработки
- **Framework**: `Echo v4` — минималистичный API с отличной производительностью и встроенной поддержкой middlewares
- **Работа с БД**: `sqlc` (генератор type-safe кода из SQL) + `pgx` (PostgreSQL Driver) — сочетают в себе type-safety, отсутствие runtime-ошибок с полями, скорость разработки и скорость работы с данными
- **Migrations**: `goose` — простой инструмент для удобной работы с миграциями


### Альтернативы

1. **Микросервисы по доменам**:
    - Отдельный сервис для событий;
    - отдельный сервис для отчётов;
    - отдельный воркер для guardrails;
    - отдельный auth‑сервис.
2. **Сервис + воркеры отдельно**:
    - HTTP‑API и фоновые процессы (guardrails, перерасчёт отчётов) разнесены по разным контейнерам/процессам.

### Последствия

Плюсы:

- **Скорость разработки**: `sqlc` избавляет от написания DTO для базы данных вручную.
- **Производительность**: cтек Go+Echo+pgx — один из самых быстрых в индустрии для I/O bound задач.
- **Надежность**: статическая типизация Go + проверка SQL на этапе компиляции (`sqlc`) минимизируют баги.
- **Простота поддержки**: код, сгенерированный `sqlc`, легко читать и отлаживать (в отличие от магии ORM).

Минусы и риски:

- **Жесткая привязка к схеме БД**: любое изменение в базе требует перегенерации кода (`sqlc generate`).
- **Отсутствие гибкости ORM**: нельзя динамически строить SQL запросы "на лету" (например, сложный фильтр по 10 полям) — придётся писать отдельные SQL запросы на каждый кейс или использовать CASE/COALESCE.
- **Бойлерплейт**: приходится маппить сгенерированные структуры БД в доменные модели.
