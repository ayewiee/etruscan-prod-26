## ADR-005: Guardrails как фоновый runner поверх каталога метрик

### Контекст

Guardrails по ТЗ:

- следят за «безопасными» метриками (ошибки, задержка и т.п.);
- должны автоматически **ставить эксперимент на паузу или откатывать** при деградации;
- опираются на те же события и решения, что и отчёты;
- работают в режиме **близком к real‑time**, но не обязаны отвечать синхронно в рамках запроса пользователя.

Важно:

- существует каталог метрик (`Metric`), включающий primitive и ratio‑метрики;
- есть общий компонент `MetricComputer`, который считает значения метрик по решениям/событиям для отчётов;
- эксперимент может иметь несколько guardrails, каждый со своим порогом, окном и действием.

Вариантов реализации два:

- встраивать оценку guardrails прямо в обработку событий `/track` или в выдачу `/decide`;
- запускать отдельный фоновой процесс, периодически пересчитывающий метрики по LAUNCHED‑экспериментам.

### Решение

Guardrails реализованы как **отдельный фоновый `GuardrailRunner`** (`internal/usecases/guardrail_runner.go`), который:

- периодически (по конфигу `GuardrailCheckIntervalMinutes`) опрашивает репозиторий экспериментов:
  - берёт все эксперименты со статусом `LAUNCHED`;
  - для каждого читает связанные guardrails;
- для каждого guardrail:
  - загружает метрику из каталога по `MetricID`;
  - через `MetricComputer` считает значение метрики за окно `[now - WindowSeconds, now]`;
  - сравнивает с порогом (direction `upper`/`lower`);
  - при срабатывании:
    - ставит эксперимент на паузу или завершает его с исходом `ROLLBACK`;
    - создаёт запись о срабатывании в `GuardrailRepository.CreateTrigger`;
    - пишет структурированный лог.

Этот runner запускается при инициализации приложения (`internal/app/app.go`) в отдельной горутине.

### Альтернативы

1. **Проверка guardrails внутри `/track`**:
   - каждый запрос на приём событий помимо валидации сразу пересчитывает связанные метрики и проверяет пороги;
   - повышает задержку `/track` и усложняет код.
2. **Проверка guardrails внутри `/decide`**:
   - каждый запрос решения запускает фоновую проверку метрик;
   - логика становится нечётко привязанной к потоку данных (guardrails завязаны на события, а не только на выдачу).

### Последствия

Плюсы:

- **Повторное использование бизнес‑логики метрик**:
  - `GuardrailRunner` использует тот же `MetricComputer`, что и отчёты;
  - это гарантирует, что guardrails считают те же числа, что видит продакт в отчётах (нет расхождений по алгоритмам).
- **Чёткая изоляция от API‑задержек**:
  - `/decide` и `/track` остаются быстрыми и не зависят от тяжёлых пересчётов метрик;
  - guardrails выполняются асинхронно с настраиваемым интервалом.
- **Простое горизонтальное масштабирование**:
  - runner можно запускать только на одном экземпляре (или использовать распределённую блокировку), не затрагивая остальные;
  - сама бизнес‑логика runner‑а не завязана на Echo/HTTP.
- **Хорошая трассируемость**:
  - при каждом срабатывании фиксируется trigger‑запись с фактическим значением метрики, порогом, окном и действием.

Минусы и риски:

- **Нечёткая граница real‑time**:
  - между отправкой деградирующих событий и срабатыванием guardrail‑а есть задержка в один интервал проверки;
  - для очень чувствительных сценариев это может быть недостаточно быстро;
  - но оно настраивается и при большой необходимости можно уменьшить интервал проверок до нескольких секунд.
- **Нужен независимый мониторинг runner‑а**:
  - если runner упадёт, guardrails перестанут срабатывать, хотя отчёты и `/decide` продолжат работать;
  - требуется мониторинг логов/метрик runner‑а.
