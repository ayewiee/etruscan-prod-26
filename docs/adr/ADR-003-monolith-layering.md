## ADR-003: Монолитный Go-сервис с разделением слоёв

### Контекст

ТЗ допускает разные варианты архитектуры A/B‑платформы: от одного сервиса до распределённой системы с несколькими микросервисами и отдельными воркерами. При этом:

- объём бизнес‑логики (флаги, эксперименты, таргетинг, события, метрики, guardrails) достаточно большой;
- время на выполнение задания ограничено;
- требуется воспроизводимый запуск через `docker compose up -d`.

Нужно было выбрать архитектуру, которая:

- проста в запуске и отладке;
- позволяет чётко разделить ответственность модулей;
- не мешает потенциальному будущему выделению подсистем (reports, ingest, guardrails) в отдельные сервисы.

### Решение

Выбран **один монолитный Go‑сервис** (`EtruscanPlatformAPI`), внутри которого реализовано разделение на слои:

- `internal/app` – сборка зависимостей (DI), конфигурация HTTP‑сервера, запуск фоновых процессов (`GuardrailRunner`);
- `internal/api` – HTTP‑слой (Echo handlers, DTO, глобальный error‑handler, валидация);
- `internal/usecases` – прикладная бизнес‑логика (decide, events, experiments, metrics, reports, guardrails);
- `internal/domain` – доменные модели и инварианты (эксперименты, флаги, метрики, guardrails, пользователи, события, bucketing);
- `internal/repository` – доступ к Postgres и Redis (sqlc‑обёртки и кеши).

Внешне система выглядит как один контейнер `API + Postgres + Redis`, запускаемый через `docker compose`.

### Альтернативы

1. **Микросервисы по доменам**:
   - Отдельный сервис для событий;
   - отдельный сервис для отчётов;
   - отдельный воркер для guardrails;
   - отдельный auth‑сервис.
2. **Сервис + воркеры отдельно**:
   - HTTP‑API и фоновые процессы (guardrails, перерасчёт отчётов) разнесены по разным контейнерам/процессам.

### Последствия

Плюсы:

- **Простой запуск**: одна команда `docker compose up -d` поднимает всё необходимое окружение.
- **Быстрая проверка**: все основные сценарии доступны через один endpoint (`:80`), не нужно разбираться с внутренними сетями нескольких сервисов.
- **Чёткое разделение слоёв в коде**:
  - доменная логика отделена от транспорта (HTTP/JSON),
  - репозитории отделены от домена,
  - фоновые задачи (guardrails) подключаются через зависимости, но не завязаны на HTTP.
- **Упрощённое тестирование**:
  - unit‑тесты для domain/usecases;
  - e2e‑тесты, поднимающие весь стек целиком.

Минусы и риски:

- **Общий процесс**: падение одного компонента внутри процесса (panic без recover) может уронить весь API, поэтому нужен аккуратный global error‑handler и panic‑recovery middleware (реализовано в `internal/app/server.go`).
- **Будущее разделение потребует дополнительной работы**:
  - для вынесения events/guardrails в отдельные сервисы надо будет выделить контракты и очереди (Kafka и т.п.), чего сейчас нет по требованиям ТЗ.
