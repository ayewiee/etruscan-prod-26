## ADR-006: Postgres + Redis для решений и событий вместо стримингового стека

### Контекст

Платформа работает с несколькими типами данных:

- **решения** (`Decision`) – запись результата `/decide`, связывающая пользователя, флаг, вариант и `decisionId`;
- **события** (`Event`) – получение данных из `/track` с типом, временем, свойствами и `decisionId`;
- **конфигурация** – флаги, эксперименты, варианты, метрики, guardrails, пользователи и группы;
- **кеши** – быстрый доступ к запущенным экспериментам, флагам и участию пользователя.

Задачи:

- гарантировать **дедупликацию** и консистентность данных для отчётов и guardrails;
- поддерживать **фильтрацию и агрегацию** по окну времени, эксперименту и варианту;
- обеспечить **простую проверяемость** (одна БД, понятные SQL‑запросы);
- уложиться в ограничения окружения (нет доступа в интернет, единый compose, ограниченные ресурсы).

### Решение

Для хранения и обработки данных выбрана связка:

- **Postgres** (основное хранилище):
  - все доменные сущности (флаги, эксперименты, варианты, метрики, guardrails, решения, события, снэпшоты, история статусов);
  - операции чтения/записи через сгенерированные sqlc‑queries (`internal/database/generated`, `internal/repository/*`);
  - дедупликация событий по `(event_type_key, client_event_id)` реализована на уровне репозитория.
- **Redis** (кеш и быстрый счётчик участия):
  - кеш запущенных экспериментов по ключу флага;
  - кеш флагов;
  - `ParticipationTracker` – ограничение числа активных экспериментов на пользователя через множества `user_active_exps:<userId>` с TTL.

Сложный стриминговый стек (Kafka, Kinesis, Flink/Spark Streaming) **не используется**, все агрегации выполняются на Postgres через `MetricComputer` по решениям и событиям.

### Альтернативы

1. **Kafka + отдельный storage для событий**:
   - `/track` пишет события в Kafka;
   - отдельный consumer складывает их в analytical storage (ClickHouse, BigQuery, Snowflake и т.п.);
   - отчёты и guardrails считают метрики по analytical storage.
2. **Kafka + Flink/Spark для онлайновых метрик**:
   - события считаются on‑the‑fly в потоковом движке;
   - результаты хранятся в key‑value хранилище, из которого отчёты и guardrails просто читают уже готовые агрегаты.

### Последствия

Плюсы:

- **Простота развёртывания**:
  - в `docker compose` достаточно поднять Postgres и Redis;
  - не нужно поднимать отдельные брокеры сообщений, координаторы и кластеры потоковой обработки.
- **Прозрачность и проверяемость**:
  - состояние экспериментов, решений и событий можно проверить обычными SQL‑запросами;
  - `MetricComputer` использует читаемые SQL‑queries, что облегчает отладку ошибок атрибуции.
- **Единая модель данных**:
  - конфигурация, decisions и events находятся в одной БД, упрощая связи и внешние ключи;
  - guardrails и отчёты используют один и тот же источник истины.
- **Ограниченное, но достаточное кеширование**:
  - Redis сокращает нагрузку на горячие запросы (`decide` и проверка участия), не усложняя модель данных.

Минусы и риски:

- **Вертикальные пределы Postgres**:
  - при большом объёме событий или высокой частоте запросов `/track` может потребоваться шардирование/партиционирование БД;
  - в рамках задания это компенсируется операционными рекомендациями (индексы, ограничения окна отчёта) в `docs/OPERATIONS.md`.
- **Отсутствие стриминговой обработки**:
  - нет возможности «в реальном времени» строить произвольные сложные метрики на лету;
  - guardrails и отчёты считают метрики по записям в Postgres с некоторой задержкой.
