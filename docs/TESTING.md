## Тестирование

Этот документ описывает, какие типы тестов есть в проекте, как их запускать и как получить отчёт по покрытию.

---

## 1. Типы тестов

- **Unit‑тесты**:
  - покрывают доменную логику;
  - примеры областей:
    - `internal/domain/models` – статусы экспериментов, переходы, роли пользователей и т.п.;
    - функции валидации значений флагов, логики экспериментов, bucketing и др.;
  - запускаются **без** базы данных и Redis.

- **E2E‑тесты**:
  - поднимают Postgres и Redis через Testcontainers;
  - выполняют миграции;
  - стартуют API‑сервис и выполняют реальные HTTP‑запросы по основным эндпоинтам.

---

## 2. Команды запуска (из `src/`)

Все команды ниже нужно запускать из каталога `src/`.

| Цель            | Команда              | Описание                                                                      |
|-----------------|----------------------|-------------------------------------------------------------------------------|
| Unit‑тесты      | `make test-unit`     | Только unit‑тесты (без БД и Docker). E2E пропускаются.                        |
| E2E‑тесты       | `make test-e2e`      | Запускает e2e‑набор с поднятием Postgres/Redis в Docker.                      |
| E2E с UI        | `make test-e2e-ui`   | E2E‑тесты через gotestsum (нужен `go install gotest.tools/gotestsum@latest`). |
| Всё сразу       | `make test`          | Unit + E2E.                                                                   |
| Покрытие (unit) | `make coverage`      | Генерирует `coverage.out` для unit‑тестов.                                    |
| HTML‑отчёт      | `make coverage-html` | Создаёт `coverage.html` для просмотра покрытия в браузере.                    |
| Cobertura отчёт | `make cobertura`     | Конвертирует `coverage.out` в `cobertura.xml` (для CI).                       |
| CI (unit)       | `make ci-unit`       | Unit‑тесты + JUnit + coverage для пайплайна (требуется gotestsum).            |
| CI (e2e)        | `make ci-e2e`        | E2E‑тесты + JUnit‑отчёт (требуется gotestsum).                                |
| Зависимости     | `make deps`          | `go mod tidy` и загрузка модулей.                                             |

Кратко:

- для быстрой локальной проверки логики → `make test-unit`;
- для полного прогона (включая сценарии с БД/Redis) → `make test` или `make test-e2e`.

---

## 3. Unit‑тесты (без БД и Redis)

- Основная концентрация:
  - доменные модели (`internal/domain/models`);
  - бизнес‑правила (например, допустимые переходы статусов эксперимента);
  - вспомогательные функции (валидация значений, выбор варианта, вычисление bucket‑ов).
- Запуск:

  ```bash
  cd src
  make test-unit
  # или
  go test -short ./internal/domain/...
  ```

Unit‑набор не требует никаких переменных окружения, Docker и внешних сервисов.

---

## 4. E2E‑тесты

- **Пакет**: `tests/e2e`.
- **Что делают**:
  - поднимают Postgres и Redis (Testcontainers);
  - применяют миграции и seed‑данные;
  - стартуют API;
  - выполняют HTTP‑запросы по сквозным сценариям:
    - health/readiness;
    - авторизация и проверка ролей (admin/viewer);
    - happy‑path фича‑флагов и экспериментов;
    - приём событий `/track` и базовая атрибуция.
- **Требования**:
  - Docker для Testcontainers;
  - запуск из `src/`, чтобы `db/migrations` были доступны (или переменная `E2E_MIGRATIONS_DIR` указывает путь к миграциям).
- **Особенности**:
  - при запуске с флагом `-short` e2e‑тесты пропускаются, так что `go test -short ./...` выполняет только unit‑тесты.

Пример запуска:

```bash
cd src
make test-e2e
```

---

## 5. Покрытие тестами

Для подсчёта покрытия используется стандартный инструмент Go.

1. Сгенерировать отчёт:

   ```bash
   cd src
   make coverage
   # или
   go test -short -coverprofile=coverage.out ./...
   ```

2. Посмотреть суммарное покрытие:

   ```bash
   go tool cover -func=coverage.out | tail -n 1
   # пример вывода:
   # total:                                  (statements) 75.3%
   ```

3. Для визуального анализа:

   ```bash
   make coverage-html
   # откройте coverage.html в браузере
   ```

Основное покрытие считается по unit‑тестам, чтобы результат был стабильным и не зависел от наличия Docker. При необходимости можно дополнить конфигурацию командой, считающей покрытие с учётом e2e‑тестов.

---

## 6. GitLab CI

Репозиторий содержит `.gitlab-ci.yml` в корне. При пуше в GitLab выполняются две основных jobs:

| Job    | Что делает                                                                                               |
|--------|----------------------------------------------------------------------------------------------------------|
| `unit` | Запускает unit‑тесты, формирует JUnit‑отчёт.                                                             |
| `e2e`  | Запускает e2e‑тесты (Docker + Testcontainers), формирует JUnit‑отчёт и отчёт по покрытию для e2e‑набора. |

В интерфейсе GitLab:

- во вкладке **Tests** видны все тесты (unit + e2e);
- показатель Coverage и отчёт Cobertura приходят из соответствующего job’а.
