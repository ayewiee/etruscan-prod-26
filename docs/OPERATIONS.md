## 1. Liveness и Readiness

Эндпоинты реализованы в `internal/app/routes.go` и доступны без аутентификации:

- **Liveness**:

  - URL: `GET /api/v1/health`;
  - назначение: показывает, что процесс сервиса жив и HTTP‑сервер отвечает;
  - типичный ответ:

    ```json
    {
      "status": "healthy"
    }
    ```

- **Readiness**:

  - URL: `GET /api/v1/ready`;
  - назначение: показывает, что приложение **готово принимать запросы**;
  - типичный ответ:

    ```json
    {
      "status": "ready"
    }
    ```

Практика использования:

- `/health` – использовать как liveness‑probe (перезапуск контейнера при недоступности);
- `/ready` – как readiness‑probe (включение/выключение трафика).

## 3. Структурированные логи (B9‑4)

Логирование построено на базе zap‑логгера:

- логгер создаётся в `internal/app/app.go` через `logger.NewZapLogger`;
- HTTP‑middleware `ZapLoggerMiddleware` (`internal/app/logger`) добавляет запись логов на каждый запрос;
- panic‑recover middleware логирует стеки паник с полями `uri`, `error` и `stack`.

Особенности:

- формат логов – структурированный (JSON‑подобные поля), удобный для парсинга в системах логирования;
- для запросов присутствуют:
  - уровень (`info`, `error`, `debug`);
  - HTTP‑метод и путь;
  - статус ответа;
  - `request_id` (генерируется `middleware.RequestIDWithConfig`);
  - время обработки.
- бизнес‑логика (`DecideUseCase`, `GuardrailRunner` и др.) добавляет:
  - значения флагов/вариантов (`value`, `variant`);
  - технические детали bucket‑ов (`participationBucket`, `variantBucket`);
  - идентификаторы экспериментов и метрик при проверке guardrails.

Примеры записей (форма зависит от конфигурации zap):

```json
{
  "level": "info",
  "msg": "HTTP request",
  "method": "POST",
  "path": "/api/v1/decide",
  "status": 200,
  "request_id": "b1f2c3d4-...",
  "latency_ms": 3.4
}
```

```json
{
  "level": "info",
  "msg": "guardrail triggered",
  "experimentId": "1c2d3e4f-...",
  "metricKey": "error_rate",
  "value": 0.12,
  "threshold": 0.05
}
```

Такая структура позволяет быстро фильтровать логи по эксперименту, метрике, пользователю и т.д.

## 4. Поведение под нагрузкой и ростом данных (B9‑6)

### 4.1 Горизонтальное масштабирование

API сервис имеет возможность горизонтального масштабирования — можно запустить несколько инстансов одновременно (за balancer'ом)
и они будут синхронизированы благодаря использованию одних PostgreSQL DB и Redis (которые тоже можно масштабировать).

### 4.2. Детали реализации

#### 4.2.1 Решения и события

Горячий путь:

- `/decide`:
  - использует кеш флагов и запущенных экспериментов в Redis;
  - всегда создаёт запись `Decision` в Postgres, но без сложных джойнов;
  - операции по чтению/записи проходят через индексы по ключевым полям (флаг, эксперимент, пользователь).
- `/track`:
  - выполняет валидацию и дедупликацию;
  - вставка событий оптимизирована под пакетную обработку.

#### 4.2.2 Отчёты и guardrails

Отчёты и guardrails работают с окнами времени:

- отчёты `/experiments/{id}/report`:
  - всегда считают метрики в заданном окне `[from, to)`;
  - это ограничивает объём данных, сканируемых при каждом запросе.
- guardrails:
  - используют скользящее окно `windowSeconds` (например, последние 5–10 минут);
  - таким образом, каждая проверка обрабатывает только свежий хвост событий.

## 5. Эффективность хранения и чтения (B9‑7)

Основные меры эффективности:

- **Индексы**:
  - на таблицах решений и событий – индексы по:
    - `experiment_id`, `variant_id`, `decision_id`, `event_type_key`, `timestamp`;
  - это обеспечивает быстрый доступ для:
    - `MetricComputer.Compute` (фильтрация по эксперименту/варианту и времени);
    - guardrails (по эксперименту и окну времени).
- **Кеши Redis**:
  - `RunningExperimentCache`:
    - кеширует эксперимент и его варианты по ключу флага, чтобы не делать запрос в Postgres при каждом `/decide`;
  - `FlagCache`:
    - кеширует флаги, используемые в `DecideUseCase` и других сценариях;
  - `ParticipationTracker`:
    - хранит только идентификаторы активных экспериментов по пользователю и не нагружает Postgres при проверке лимитов участия.

В результате:

- частые операции (`decide`, `track`) используют минимальный набор индексов и кешей;
- тяжёлые операции (`report`, guardrails) ограничены окнами времени и тоже опираются на индексы.
